bugfolder (Dec 31 2020 at 18:48):

Following up on my module title question from yesterday, this is Ubercart-specific (hi @Laryn); but is generally applicable to contrib modules that extend other contrib modules (e.g., Webform add-ons).

Having just finished my second Ubercart module port (and more on the way), I'm wondering about standards for Ubercart-related module naming in Backdrop.

Most of the Backdrop Ubercart contrib modules are currently named "Ubercart Something". Whereas over on the Drupal side, most are just called "Something", and a few are called "UC Something". This makes me wonder:

1) For modules created solely for Backdrop Ubercart (no D7 port), is there a preferred naming scheme?

2) For modules ported from D7, should the name be exactly what it was in D7 (modulo capitalization standardization), or should it be adjusted to follow a preferred/consistent naming scheme (i.e., add or remove "Ubercart" or "UC" to/from the name)?

Then there's the question of which package the contrib module should belong to.

3) Should the package name for an Ubercart contrib module be one of the non-core Ubercart package names already defined by the Ubercart modules (i.e., "Ubercart - extra", "Ubercart - fulfillment", "Ubercart - payment"), or a specific separate name (e.g., "Ubercart - contrib"), or are there no particular recommendations?

4) Whatever the answers to 1–3 are, would it be a good idea to document that information on the Ubercart project Wiki page (which I'd be happy to draft)?

Phil Ward (Jan 05 2021 at 23:04):

@docwilmot if github is doing things the way I think they are with webhooks, then yes you should be able to set it up on your local dev.

From my understanding, you have a listener that listens for intents sent by an endpoint. If it hears that intent, it does something with it.

As long as the endpoint can reach the internet (not blocked by an apache .htaccess password login for example) then anything can be configured as an endpoint.

My terminology might be a little off but in essence, instead of sending a file with information like XML or json, you send a machine name (intent) to the listener. If the listener sees (hears) it, it checks authenticity through the token and routes it to the appropriate account based on the API key.

It's a way to send real-time information over the wire without having to send a bunch of files every time something needs to be done.

I want to say it can be asynchronous as well, but I'm fuzzy on that part.

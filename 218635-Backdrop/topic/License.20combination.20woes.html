bugfolder (Dec 20 2020 at 02:59):

I'm working on a port of the 7.x-2.x branch of the D7 Google Auth module, which is GPL2. It uses the Google API PHP Client, which is Apache 2. In D7, the Google library is separate and resides in the libraries folder (and depends on the Libraries module, which is deprecated in BD). In Backdrop, we are supposed to include such libraries within the module, which thereby raises the problem.

According to here, I can include Apache 2.0 code in a GPL 3 package, but not GPL 2.

And according to here I can't include GPL2 code (the original module) in a GPL 3 package.

So, if I'm understanding correctly, the only way forward would be to persuade the original developer to re-release the 7.x-2.x module under GPL3, base the BD port off of that, and then I could include the Google module with the Apache licensed code.

Is this correct? Or are there other approaches to consider?

Can I bundle Apache 2.0 licensed components in a GPL 3 licensed project?
I am creating a project that I would like to open source and license under the GPL 3 License. The project will come with a bundled version of an open source component which is licensed under the Ap...
Including GPLv2 code in GPLv3 code
I would like to use some code that was released under GPLv2 in my project. I released my project with the GPLv3 license. Can I do this, or do I have to downgrade to GPLv2?

oadaeh (Jason Flatt) (Dec 20 2020 at 03:21):

Have you read the README.md file for the Libraries API module? It suggests a possible option by using Backdrop core equivalent functions. I haven't gone down that path myself, so I don't know if that would help or not.

oadaeh (Jason Flatt) (Dec 20 2020 at 03:21):

Another option is to specify in the README.md file of the module you're porting that the person using needs to download the library themselves and place it somewhere of your choosing. I realize that is taking a giant step back to the days when there was no consistent way of handling libraries in Drupal.

Heads-up: I have multiple tabs open in my browser, where I have drafted several new issues from the various topics discussed during the weekly UX/design meeting on Dec. 31. (I regret having missed that meeting, but it was on 10pm on the NYE for me).
Anyway, I just wanted to make a note with regards to the discussion around layouts, blocks, visibility conditions, and handling paths (the part tha happens ~50min into the call: YouTube - YouTube). This has come up before in the past, but it seems to me that people either do not realize that there are 2 ways to approach things, or they keep forgetting it until reminded:

  • @jenlampton and @StPaulTim seem to prefer or have gotten used to using a single layout for a path, and then use multiple similar blocks. Visibility conditions are added to each block, in order to define what each user will see (think logged-in vs anon users)

  • @Phil Ward on the other hand seems to prefer having 2 separate layouts for the same path, with each different block added to the respective layout (one for logged-in + another for anon users). In this case, the visibility conditions are added to the layouts instead of the blocks.
    ...I'm not saying that one way is better than the other, because:

  • On the one hand, using a single layout for a path, and then placing 2 (or more) blocks which are basically the same thing but with different visibility conditions may work, but it may not always be ideal. Over time, we have improved UX by adding features like visibility condition preview/summary for each block + better admin block titles. All that may help somewhat, but you may still end up having to deal with huge lists of blocks within the same layout.

  • On the other hand, if you use 2 layouts + visibility conditions on the layouts instead of the blocks, it makes it relatively easier to know which layout you need to edit, and each layout is kept "tidy" (small amount of blocks to deal with). The downside is that you will need to duplicate blocks that should be visible in both layouts. When you configure a "common" block in one layout (logged-in users), you then need to repeat the configuration on the other layout too (the one for anon users). Instead of switching between layouts, one can have them both open on separate browser tabs. This may work, but still not ideal having to repeat configuration.
    ...
    ...the reason why I've taken the time to write all the above, is that @jenlampton keep mentioning that this is a bug, while @Phil Ward kept trying to say that he thinks not ...I agree with @Phil Ward with regards to it NOT being a bug, but not necessarily that it is the best way to handle things ...I think that somewhere in that discussion you all were talking about similar but different things ...and I think that terminology confusion occurred at some point :sweat_smile: ...anyway, it's really hard to put all thoughts down in words, so I would love to revisit that discussion during the next UX meeting, and clarify certain things.

Phil Ward (Jan 12 2021 at 20:55):

I was thinking about this discussion the other day and I feel like "both should work". I think the use-case of one over the other depends on the person and the situation. I understand it makes documentation/tutorials more confusing when there are multiple ways to accomplish the same task so that might need to be a consideration.

Ah, and I just remembered I ran into the problem of keeping "panes" synced in panels. I always envisioned a way to make a pane (block in this case) a default for a given variant (layout) and any changes to it would propagate to the other variants (layouts). Kind of like views. It could be done, but adds an additional layer of complexity.

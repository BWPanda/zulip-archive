oadaeh (Jason Flatt) (Dec 13 2020 at 16:47):

One of my pet peeves (maybe the only one, but certainly the biggest) is the lack of adequate error checking and reporting done in Drupal (and by extension, Backdrop).

I suspect you are correct with the first error, but for the second one, more error checking and reporting could certainly be done there.

The code doesn't do any basic file checks at all, and the call to file_put_contents() actually hides the actual error (with the "@" in the 6th line below) that would be recorded in watchdog, making that code not only insufficient, but also less than helpful by removing any information that could be used to fix the problem.

  public function write($name, array $data) {
    // Ensure that the config name is included in the written file.
    $data = array_merge(array('_config_name' => $name), $data);
    $data = $this->encode($data) . "\n";
    $file_path = $this->getFilePath($name);
    $status = @file_put_contents($file_path, $data);
    if ($status === FALSE) {
      throw new ConfigStorageException('Failed to write configuration file: ' . $this->getFilePath($name));
    }
    clearstatcache(FALSE, $file_path);
    return TRUE;
  }

I don't know if adding that would help a new user, but I suspect if someone is getting a file permission error, it is because they put the files there, and didn't use a one-click type of install process, so they should know how to fix it with an adequete error message.

cellear (Luke McCormick) (Jan 13 2021 at 23:18):

I have a crazy thought about config management.

CMI was intended to make some things easier. Because configuration is stored in the file system instead the database, it's easier to do things that were difficult to do when configuration was stored in the database --- notably, track configuration changes in a VCS.

The downside is, it makes some things harder. If you're doing something that involves moving the database then CMI just means that there is an additional piece to worry about. And that piece is somewhat brittle and finicky, as discussed above.

So...what if we did both? Let configuration be stored not just in files, but also (perhaps optionally) in the database? It could be super simple, just a table with two fields, for the title and the contents of each config file. One row for each file.

That way, if you move the database, you don't have to move the config files. You could use Backup&Migrate unmodified, for example. It could be just a third config source -- Active, Staging, and DB.

Would anybody besides me find this appealing?
